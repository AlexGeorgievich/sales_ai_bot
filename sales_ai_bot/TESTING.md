# Инструкция по тестированию проекта Sales AI Bot 🧪

В проекте реализовано два уровня автоматизированного тестирования:
1. **Бэкенд-тесты (Pytest)** — быстрые изолированные юнит- и интеграционные тесты API, Circuit Breaker и кэширования.
2. **Фронтенд E2E-тесты (Playwright)** — кросс-браузерные сквозные тесты интерфейса лендинга, валидации форм, виджета чат-бота и полной интеграции с базой данных и админ-панелью.

---

## 1. Бэкенд-тестирование (Pytest)

Бэкенд-тесты проверяют бизнес-логику API, кэширование и паттерн предохранителя. Для обеспечения независимости от внешнего окружения тесты используют:
* Изолированную асинхронную СУБД SQLite в памяти (`sqlite+aiosqlite:///:memory:`).
* Эмулятор Redis в оперативной памяти (`fakeredis`), имитирующий реальные TTL, кэш-ключи и статистику.
* Моки запросов к API GigaChat.

### Состав тестов бэкенда:
* `test_cache_service.py` — проверка сохранения ответов, разделения кэш-записей по типам клиентов (B2B/B2C), учета контекста диалогов (хэш истории) и отслеживания статистики попаданий (hits/misses).
* `test_circuit_breaker.py` — проверка перехода предохранителя между состояниями `CLOSED`, `OPEN` и `HALF-OPEN` при отказах внешнего API, а также корректность выдачи заглушек (fallback-ответов).
* `test_api.py` — проверка работоспособности health-check эндпоинтов, очистки кэша и корректности обработки входящих сообщений.

### Команды запуска бэкенд-тестов:
Активируйте виртуальное окружение в корне репозитория:
```bash
. .\venv\Scripts\Activate.ps1
```
Перейдите в папку `sales_ai_bot` и запустите тесты:
```bash
# Обычный запуск бэкенд-тестов (игнорируя фронтенд)
python -m pytest -v --ignore=tests/test_frontend.py

# Запуск с отчетом о покрытии кода (coverage)
python -m pytest --cov=app --cov-report=term-missing --ignore=tests/test_frontend.py
```

## 2. Фронтенд-тестирование (Playwright)

Фронтенд-тесты проверяют интерактивный виджет чат-бота, адаптивность лендинга, валидацию полей форм на клиенте и интеграционную связку с админ-панелью.

### Требования для запуска:
* Установленный Chromium браузер от Playwright.
* Запущенное локально веб-приложение в Docker (по умолчанию тесты обращаются к `http://localhost` и `http://localhost:8501`).

### Перечень E2E сценариев (19 тестов):

1. **`test_landing_page_load`** — Проверка загрузки лендинга, шапки с логотипом, H1-заголовка и Hero-секции.
2. **`test_landing_navigation_scroll`** — Проверка плавной прокрутки страницы к соответствующему якорю (секции Услуги) при клике в навбаре.
3. **`test_landing_mobile_hamburger_menu`** — Проверка мобильной адаптивности: на ширине `375px` (iPhone 12) появляется кнопка бургера, при клике открывается меню.
4. **`test_form_tabs_switching`** — Переключение вкладок B2B («Для бизнеса») и B2C («Для частных лиц») с проверкой скрытия/показа форм.
5. **`test_b2b_form_validation_required_fields`** — Проверка HTML5 валидации: поле Имя в форме B2B является обязательным к заполнению.
6. **`test_b2b_form_validation_email`** — Валидация некорректного формата email адреса в форме B2B (например, `invalid-email`).
7. **`test_b2b_form_validation_privacy`** — Проверка обязательности согласия с политикой обработки персональных данных для B2B.
8. **`test_b2b_phone_mask`** — Проверка JS-маски телефона для B2B: при вводе цифр они форматируются в `+7 (999) 123-45-67`.
9. **`test_b2c_form_validation_required_fields`** — Проверка HTML5 обязательности поля Имя для формы B2C.
10. **`test_b2c_form_validation_email`** — Валидация формата email адреса в форме B2C.
11. **`test_b2c_form_validation_privacy`** — Проверка обязательности согласия с политикой персональных данных для B2C.
12. **`test_b2c_phone_mask`** — Проверка JS-маски телефона для B2C.
13. **`test_b2b_submission_payload_with_mock`** — Перехват POST-запроса B2B к API `/chat/leads` с валидацией структуры JSON (ФИО, телефон, продукт, тип клиента).
14. **`test_b2c_submission_payload_with_mock`** — Перехват POST-запроса B2C к API `/chat/leads` с валидацией структуры JSON.
15. **`test_chat_widget_toggle_ui`** — Открытие чат-виджета кликом по кнопке-кружку и его закрытие по крестику.
16. **`test_chat_widget_welcome_message`** — Автоматическое появление приветственного сообщения от ИИ-ассистента при открытии чата.
17. **`test_chat_widget_send_message_mock`** — Отправка вопроса в чат-виджет, появление индикатора печатания бота (typing) и отображение ответа.
18. **`test_e2e_integration_b2b_form_to_streamlit_admin`** — Сквозной E2E B2B сценарий: отправка формы на реальном лендинге -> запись в PostgreSQL -> автоматический переход в Streamlit-админку -> переключение на вкладку «Лиды» -> проверка наличия созданного лида.
19. **`test_e2e_integration_b2c_form_to_streamlit_admin`** — Сквозной E2E B2C сценарий: отправка формы -> переход в админку -> проверка лида.

## 3. Варианты получения отчетов по тестам

### Вариант A: Классический консольный отчет
```bash
python -m pytest tests/test_frontend.py -v
```

### Вариант B: Локальный HTML-отчет (pytest-html)
Генерирует один автономный файл-отчет `report.html`, содержащий общую статистику прохождения:
```bash
python -m pytest tests/test_frontend.py --html=report.html --self-contained-html -v
```
*Открыть отчет можно командой `Start-Process report.html`.*

### Вариант C: Интерактивный Allure-отчет (с графиками, шагами и видео)
Предоставляет детальный дашборд с круговыми диаграммами, распределением тестов, скриншотами и видеозаписями тестов.

1. **Запустите тесты для сбора результатов**:
   ```bash
   make test-allure
   ```
   *(Команда под капотом: `pytest tests/ --alluredir=allure-results --screenshot=on --video=on -v`)*

2. **Запустите Allure веб-сервер для просмотра отчета**:
   ```bash
   make allure-serve
   ```
   *(Команда под капотом: `allure serve allure-results`)*

#### Особенности отчетов Allure в проекте:
* **Автоматические заголовки из docstring:** В [tests/conftest.py](conftest.py) настроена автоподстановка первой строки описания теста из кода в качестве заголовка в Allure.
* **Видеозаписи (`video_walkthrough`):** К каждому тесту прикрепляется запись прохождения.
* **Скриншоты (`final_screenshot`):** В конце каждого теста делается снимок экрана.
* **Отсутствие серых полей:** Благодаря аргументам запуска браузера `--disable-gpu` и `--disable-software-rasterizer` видеозаписи не имеют дефектов отрисовки в headless-режиме Windows.
