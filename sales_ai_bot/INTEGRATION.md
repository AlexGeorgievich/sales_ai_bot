# Документация по интеграции и архитектуре

## 1. Интеграция с GigaChat API

Для обработки естественного языка используется официальный асинхронный SDK `gigachat`.

- **Авторизация:** OAuth 2.0 через `GigaChat API Key` (хранится в `.env`).
- **Обход SSL:** В режиме `DEBUG=True` отключена строгая проверка SSL-сертификатов для локального тестирования.
- **Контекст:** Бот отправляет в API системный промпт + последние 20 сообщений из истории диалога, что позволяет поддерживать связную беседу.

## 2. Интеграция с ProxyAPI

Для безопасности и балансировки нагрузки используется ProxyAPI.

- **Ротация IP:** В `app/core/proxy_manager.py` реализован пул прокси. Запросы к GigaChat распределяются по схеме Round-Robin, что предотвращает IP-бан при 1000+ одновременных сессиях.
- **Отказоустойчивость:** Если прокси недоступен, запрос уходит напрямую.

## 3. Продвинутая маршрутизация запросов (Request Routing)

В `app/core/gigachat_service.py` реализован трехуровневый роутинг:

1. **Уровень 1 (Кэш):** Если вопрос типовой и уже есть в Redis (с учетом контекста), ответ отдается за <0.01 сек без обращения к ИИ.
2. **Уровень 2 (GigaChat + Circuit Breaker):** Если кэша нет, запрос идет в GigaChat. Паттерн Circuit Breaker (`app/core/gigachat_service.py`) отслеживает ошибки. При 5 подряд сбоях API "открывается", и бот мгновенно отдает fallback-ответ, не вешая пользователя.
3. **Уровень 3 (Fallback/Менеджер):** Если ИИ недоступен или вопрос слишком сложный, бот переводит диалог на менеджера.

## 4. Управление сессиями пользователей (Session Management)

- **Идентификация:** При первом заходе виджет генерирует уникальный `session_id` (UUID) и сохраняет его в `localStorage`.
- **Хранение контекста:** История сообщений пишется в PostgreSQL (`app/db/models.py` -> таблица `messages`). Для контекста ИИ берутся только последние N сообщений, чтобы не превысить лимит токенов.
- **Изоляция:** Сессии жестко привязаны к `user_id` и `session_id`, что исключает пересечение диалогов разных пользователей.
